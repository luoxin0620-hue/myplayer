# 设计与架构评分 Rubric

当请求涉及多个类、多个文件、模块边界、依赖关系、架构设计、设计模式、耦合或职责分配时，使用本 rubric。

## 评分原则

- 这是设计层评分，不是实现细节评分。
- 只有在可见代码足以支撑判断时才评分。
- 如果证据不足，明确说明“只能做局部观察，不能做完整架构判断”。
- 分数是教学辅助，不是精确测量。

## 等级参考

- `90-100`: 结构清晰
- `75-89`: 整体合理
- `60-74`: 正在成形
- `40-59`: 设计脆弱
- `0-39`: 需要重构基础结构

## 默认维度与权重

1. 类职责与单一职责: `20`
2. 接口设计与封装边界: `15`
3. 模块划分与内聚性: `15`
4. 耦合度与依赖方向: `20`
5. 可扩展性与变化隔离: `15`
6. 设计模式使用是否恰当: `10`
7. 可测试性与替换成本: `5`

如果某些维度完全不可观察，跳过并把注意力集中在能判断的部分。

## 观察什么

### 类职责与单一职责

- 一个类是否承担了过多不同性质的工作。
- 类是否同时处理业务规则、IO、状态协调、展示或资源管理。
- 是否容易用一句话解释“这个类负责什么”。

### 接口设计与封装边界

- 公共接口是否暴露过多内部细节。
- 是否把本应私有的状态和实现细节泄漏给外部。
- 接口是否表达了正确的约束、所有权和使用方式。

### 模块划分与内聚性

- 同一模块中的代码是否围绕一个清晰目的组织。
- 是否把本该一起变化的代码放在了一起。
- 是否出现“为了调用方便把无关功能堆到一个模块里”的情况。

### 耦合度与依赖方向

- 模块或类之间是否过度直接依赖。
- 是否存在双向依赖、循环依赖或层次反转。
- 高层策略是否被底层实现细节牵着走。

### 可扩展性与变化隔离

- 未来加一个新能力时，是否需要改很多不相关的地方。
- 是否存在明显的 shotgun surgery 风险。
- 变化点是否被封装在少数边界内。

### 设计模式使用是否恰当

- 是否为了“用了某种模式”而引入额外复杂度。
- 是否缺少最基本的抽象，导致代码难以替换和扩展。
- 模式是否真的解决了耦合、扩展或职责问题。

### 可测试性与替换成本

- 一个模块是否容易被替换、模拟或单独验证。
- 是否存在对全局状态、具体实现或环境的硬绑定。

## 常见设计层信号

以下信号通常意味着设计层有问题：

1. 一个类同时负责业务、状态、IO 和协调
2. 调用链很长，而且到处传递同一批依赖
3. 为了增加一个功能，需要改很多看起来不相关的类
4. 模块之间频繁相互包含、相互调用
5. 公共接口暴露太多 setter、可变状态或内部结构
6. 为了复用而继承，但实际上只是想共享一点实现
7. 抽象层很多，但每层都很薄，增加了理解成本

## 输出建议

当你指出设计问题时，优先解释：

1. 现在的职责边界在哪里变得模糊
2. 哪条依赖路径导致了高耦合
3. 未来哪类修改会因此变得困难
4. 可以先做哪一个最小设计调整

尽量优先建议“小而正确的结构优化”，而不是大而空的架构重写。
