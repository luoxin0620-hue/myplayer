# 架构、耦合与类设计观察指南

当用户希望得到更深层反馈时，用本文件帮助识别类设计、模块边界和架构层问题。

## 先判断证据是否足够

只有当你能看到以下至少一类证据时，才做较强的架构判断：

- 多个类之间的调用关系
- 多个模块或目录之间的边界
- 明显的依赖流向
- 关键核心类的职责分布
- 用户明确提供的架构背景

如果只能看到一个短函数或一个很小的文件，只能做局部设计点评，不要假装看懂了整个系统。

## 重点识别的问题

### God Object / God Class

信号：
- 类的方法很多，而且分属不同职责
- 类同时持有大量依赖
- 很多功能都必须经过这个类中转

指导意见：
- 先按职责拆分，而不是按代码行数拆分
- 识别“协调者”“状态持有者”“执行者”三类角色
- 把最容易变化的职责优先抽离

### 高耦合

信号：
- 一个模块改动时，总会牵连很多其他模块
- 类之间通过具体实现而不是稳定接口交互
- 调用方知道被调用方太多内部细节

指导意见：
- 缩小公共接口
- 面向稳定抽象编程，而不是直接依赖细节
- 隔离最常变化的实现

### 低内聚

信号：
- 同一个类中的方法操作的是不同字段集合
- 一个模块里混杂了不相干的概念
- 无法用一句清晰的话概括这个类/模块的职责

指导意见：
- 先写出“这个类只负责什么，不负责什么”
- 把一起变化、一起被理解的逻辑放在一起
- 拆掉“为了方便放一起”的无关功能

### 边界泄漏

信号：
- 上层代码频繁操作下层内部数据结构
- 领域逻辑夹杂框架、IO 或界面细节
- 公共头文件或公共模块暴露过多实现依赖

指导意见：
- 收紧边界，只暴露必要能力
- 用更小、更稳定的接口隔离内部变化
- 避免把实现细节传播到更高层

### 双向依赖 / 循环依赖

信号：
- A 依赖 B，同时 B 也依赖 A
- 模块必须互相知道对方内部细节
- 稍微改动结构就引发 include 或构建问题

指导意见：
- 找出谁才是真正的拥有者和决策者
- 提取稳定接口或中间抽象打断环
- 优先让依赖朝向更稳定、更抽象的方向流动

### 模式滥用

信号：
- 类和接口很多，但业务很简单
- 为了套某种模式，增加了不必要的跳转层
- 抽象存在，但没有带来替换性、测试性或扩展性收益

指导意见：
- 先问“这个抽象解决了什么真实变化点”
- 如果回答不清，就优先简化
- 模式是手段，不是目标

### 缺少必要抽象

信号：
- 新增一个变体时需要复制粘贴旧逻辑
- 条件分支随着功能扩展越来越长
- 依赖无法替换，测试困难

指导意见：
- 在重复变化点出现后，再抽象
- 优先抽象变化轴，而不是抽象所有东西
- 保持抽象数量少但语义清楚

## 类设计点评模板

当你点评类设计时，优先覆盖这 4 个问题：

1. 这个类的核心职责是什么
2. 这个类现在混入了哪些不该由它负责的事
3. 它和哪些类或模块耦合过深
4. 最小可执行的拆分或收口动作是什么

## 架构点评模板

当你点评架构时，优先覆盖这 4 个问题：

1. 当前边界是怎样划分的
2. 哪些依赖方向不合理
3. 哪些变化点没有被隔离
4. 下一步最值得做的结构调整是什么

## 面向初学者的建议方式

- 不要一上来就建议复杂模式
- 优先建议职责收敛、边界收紧、接口变小、依赖变少
- 优先解释“为什么这样更容易改、更容易测、更不容易牵一发而动全身”
- 如果要举模式例子，先给直白解释，再提模式名
