# 通用评分 Rubric

使用这个 rubric 输出“教练式评分”，而不是简单的通过/不通过式审查。

## 评分区间

- `90-100`: 很强
- `75-89`: 扎实
- `60-74`: 仍在成长
- `40-59`: 比较脆弱
- `0-39`: 基础能力需要补强

把这个分数当成方向性参考，而不是精确科学测量。证据不充分时，分数按最接近的 5 分取整。

## 默认维度与权重

只对当前可见代码中能够判断的维度进行评分。

1. 可读性与清晰度: `20`
2. 命名与意图表达: `15`
3. 结构与拆分: `15`
4. 状态与数据所有权: `15`
5. 错误处理与边界情况: `10`
6. 语言惯用法使用情况: `10`
7. 可测试性与可修改性: `10`
8. 注释与文档: `5`

如果某个维度无法观察到，不要假装给它评分，而是把注意力自然转移到其余可判断维度上。

## 观察要点

### 可读性与清晰度

- 控制流程是否容易跟踪。
- 关键逻辑是否被埋在过于密集的表达式中。
- 排版和布局是否有助于阅读。
- 当重复已经遮蔽意图时，是否进行了合理收敛。

### 命名与意图表达

- 变量、函数和类型是否能表达用途。
- 布尔命名是否像状态或问题。
- 临时性命名是否混入了正式逻辑。
- 魔法数字和难懂缩写是否被控制住。

### 结构与拆分

- 函数是否只承担一个连贯职责。
- 参数列表是否没有混入不相关关注点。
- 辅助函数是否在有助于简化主流程时被提取出来。
- 类和模块的边界是否清晰。

### 状态与数据所有权

- 可变状态是否受控且容易推理。
- 所有权和生命周期是否清楚。
- 数据传递方式是否避免了隐藏耦合。
- 全局状态和隐藏副作用是否被避免，或至少有充分理由。

### 错误处理与边界情况

- 非法输入、空情况和失败路径是否被考虑到。
- 当假设被打破时，代码是否会以可预期方式失败。
- 返回值和异常路径是否没有被忽略。

### 语言惯用法使用情况

- 代码是否遵循该语言常见的写法习惯。
- 当标准库能提升清晰度时，是否优先使用标准库而不是重复造轮子。
- 是否避免了“为了显得高级”而使用复杂特性。

### 可测试性与可修改性

- 逻辑是否能在不做大量环境准备的情况下被验证。
- 在合适场景下，是否采用了纯函数或隔离良好的函数。
- 紧耦合是否被尽量压低。
- 代码是否容易扩展，而不需要大面积重写。

### 注释与文档

- 注释是否解释“为什么”，而不仅仅是“做了什么”。
- 公共 API 和不明显的约束条件是否有文档说明。
- 注释是否没有沦为噪音或简单复述代码。

## 反馈优先级

优先指出以下几类问题：

1. 用户可以在很多地方一起改掉的坏习惯
2. 会让未来修改越来越困难的代码形态
3. 掩盖了更简单方案的不符合语言习惯的写法
4. 已经实质影响可读性的风格问题

避免把大部分篇幅浪费在：

- 琐碎的格式细节
- 影响很低的个人风格偏好
- 没有证据支撑的性能猜测

## 教学式反馈模板

对于每个重要问题，尽量按下面结构来写：

1. `薄弱点是什么`
2. `为什么重要`
3. `怎样改进`
4. `更好的示例`

示例保持简短。优先给出用户可以真实落地的最小改写版本。
