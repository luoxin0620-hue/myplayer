# C 与 C++ 初学者常见习惯问题

当评估 C 或 C++ 代码、并且目标是给出学习型质量反馈时，加载这个文件。

## 重点关注区域

优先关注这些点，因为它们更能培养长期有效的好习惯：

1. 资源所有权与生命周期
2. const 正确性
3. 头文件中的接口设计
4. 有意义的命名
5. 避免不必要的原始指针
6. 当标准工具足够时，避免手动内存管理
7. 清晰地分离声明与实现
8. 当标准库类型和算法能提升清晰度时优先使用它们

## 常见薄弱习惯与更好的模式

### 过度暴露可变状态

薄弱习惯：
- 没有明确理由却直接暴露 public 数据成员
- 随意返回可变引用

应引导：
- 隐藏实现细节
- 暴露更能表达意图的操作函数
- 让无效状态更难被表示出来

### 缺少 const 正确性

薄弱习惯：
- 明明不修改状态的成员函数却没有标记 `const`
- 只读参数却用非常量引用传递

应引导：
- 用 `const` 明确表达意图和安全使用方式
- 对只读且复制成本高的对象优先使用 `const T&`
- 对体积小、语义简单的类型直接按值传递

### 头文件过重

薄弱习惯：
- 没有必要却把实现逻辑放进头文件
- 引入了超过实际需要的头文件
- 把私有依赖泄漏到公共头文件中

应引导：
- 让头文件只聚焦接口
- 在合适时使用前向声明
- 把非平凡实现移到 `.cpp` 文件中

### 不必要的原始资源所有权管理

薄弱习惯：
- 在普通应用代码中直接写 `new` 和 `delete`
- 手动清理逻辑散落在多个分支里

应引导：
- 优先使用自动存储期对象、`std::string`、`std::vector` 和智能指针
- 使用 RAII，让清理随着对象生命周期自动发生

### 在 C++ 中写得过于 C 风格

薄弱习惯：
- 普通文本处理仍然使用字符数组缓冲区
- 明明标准算法更清晰，却还是手写循环
- 明明常量或内联函数更合适，却继续使用宏

应引导：
- 当现代标准库设施能提升安全性和清晰度时优先使用它们
- 使用语言特性是为了简化代码，而不是为了显得“现代”

### 指针表达的所有权不清

薄弱习惯：
- 使用原始指针却没有明确所有权语义
- 工厂函数返回指针，却不说明由谁负责释放

应引导：
- 对非空、借用式访问优先使用引用
- 当所有权转移是重点时使用智能指针
- 当原始指针不可避免时，在接口中明确说明所有权

### 函数过于臃肿

薄弱习惯：
- 一个函数同时负责解析、校验、状态修改和输出

应引导：
- 按职责拆分
- 辅助函数按它们完成的任务命名
- 让主流程从上到下都容易阅读

### 枚举和常量使用薄弱

薄弱习惯：
- 魔法数字
- 会污染命名空间的非作用域枚举

应引导：
- 优先使用具名常量
- 当含义属于特定领域且需要作用域时，使用 `enum class`

### 注释只是在复述显而易见的内容

薄弱习惯：
- 注释写成“i 自增”或“打开文件”这种对代码逐句翻译

应引导：
- 注释应说明意图、不变量、权衡或令人意外的行为
- 删除那些只是重复代码内容的注释

## 对 C++ 的输出侧重点

在评估 C++ 代码时，额外关注以下方面：

1. 接口是否清楚表达了所有权
2. 头文件是否只暴露了必须暴露的内容
3. 标准库设施是否能让代码更简单清晰
4. 命名和拆分方式是否让代码更容易扩展

## 改写建议原则

当你给出改写建议时：

- 优先推荐更简单、接近 C++17 日常风格的改进，而不是模板味很重的高级写法
- 除非代码确实需要，否则不要急着推荐移动语义、元编程或复杂抽象
- 如果存在多种修复方式，优先展示最能帮助用户建立扎实基础的那一种
